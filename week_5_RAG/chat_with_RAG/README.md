# RAG-чат по книге «Алиса в Стране чудес» — финальный этап

Интерактивный stateful-чат поверх готового RAG-движка. Не разовые вопросы, а осмысленный
диалог: агент удерживает **цель разговора**, **зафиксированные термины и ограничения** и
**историю переписки**, продолжая на каждом шаге искать в базе `rag_wonderland.db` и отвечать
строго по первоисточнику с цитатами и источниками.

Ничего в RAG-движке не переписано. Весь поиск/реранк/структурный ответ импортируется как есть
из `week_5_RAG/reranking_and_rewrite` (который сам переиспользует `week_5_RAG/request_to_RAG`).
Концепт «история сообщений + отдельный State-объект, переживающий обрезку окна» взят из
`week_3_stateful_agent` (там это `MemoryEngine` со слоями short-term / working / invariants).

## Шаг 1 — Память задачи (`TaskState`)

`TaskState` — датакласс с тремя полями, живущий **отдельно** от истории сообщений (см.
`task_state.py`):

- `goals` — что сейчас является целью диалога. **Первый элемент — сквозная цель всего
  разговора** (например, *«Learn everything about the Cheshire Cat»*), дальше 1–4 подцели под
  текущий фокус.
- `constraints_and_terms` — зафиксированные ограничения и термины (*«do not mention the Queen of
  Hearts»*, *«'the accused' = the Knave of Hearts»*).
- `user_clarifications` — что пользователь уже уточнил по ходу беседы.

**Как обновляется.** Перед каждым основным запросом к DeepSeek запускается отдельный **лёгкий**
LLM-вызов (`update_task_state`, `temperature=0`, JSON-выход): он сворачивает новое сообщение
пользователя в текущее состояние. Это и есть «Перед каждым запросом к DeepSeek обновляй
TaskState» из задания.

**Почему память действительно «липкая» (две страховки, не только промпт):**

1. *Промпт апдейтера* обязан сохранять сквозную цель первым элементом и запрещает опустошать
   `goals` на ходах вида «summarize» / «switch focus».
2. *Код-гарантии* (детерминированные, не зависят от настроения модели):
   - `chat_agent.py` пиннит первую установленную сквозную цель и на каждом ходу гарантирует, что
     она остаётся первой в `goals` (переживает «смену фокуса», «подведи итог» и обрезку окна);
   - `task_state.py` не даёт `goals` стать пустым и никогда не роняет ранее заданное
     ограничение (constraints — sticky).

Именно поэтому окно истории намеренно маленькое (`HISTORY_WINDOW = 8`, т.е. ~4 хода): к
12-му сообщению «сырой» истории первого сообщения уже нет в окне, но цель по-прежнему цела —
её держит `TaskState`, а не история. Это и проверяет автотест.

## Шаг 2 — Интеграция RAG в контекст ЧАТА

На каждое сообщение (`ChatAgent.ask`, `chat_agent.py`):

1. **Апдейт `TaskState`** (лёгкий LLM-вызов, до всего остального).
2. **Context-aware rewrite.** Текущий вопрос + последние 4 хода пользователя подаются в
   переиспользованный `rewrite_query`, чтобы он разрешил местоимения по истории:
   `«And how does he disappear?»` → `«How does the Cheshire Cat disappear?»`.
3. **Поиск + порог + реранк.** По переписанному запросу — `retrieve_chunks_advanced` (top-10
   cosine → жёсткий порог → cross-encoder rerank в top-3), без модификации функции.
4. **Сборка промпта и генерация.** В системный промпт упаковываются: структурный RAG-промпт из
   прошлого этапа, блок активных ограничений, `TaskState`, окно истории (реальными
   `user/assistant`-сообщениями) и свежий контекст этого хода. Модель обязана ВСЕГДА выводить
   три блока — `## Answer`, `## Quotes Used`, `## Sources` — даже на коротких уточнениях.

### Релевантность в чате: два слоя вместо одного порога

В одноразовом QA-эвале прошлого этапа порог был `0.60`. В чате он **намеренно снижен до `0.50`**
(`CHAT_SIMILARITY_THRESHOLD`). Причина честная: короткие реплики-уточнения («and his grin?»)
дают более низкий cosine даже когда они строго по теме — у `nomic-embed-text` реальные
внутрикнижные чат-запросы попадают в 0.53–0.60 и перекрываются с off-topic band. Порог `0.60`
рубил бы законные follow-up'ы. Поэтому в чате порог — только «ловушка мусора» (не тратить API на
совсем несвязный ввод), а точную релевантность держат два других слоя, оставшихся в полной силе:

1. **Cross-encoder rerank** отбирает лучшие 3 чанка из прошедших порог.
2. **Структурный soft-refusal**: если в отобранных чанках ответа всё же нет, модель по промпту
   пишет в `## Answer` «в контексте нет ответа», а в `## Quotes Used` / `## Sources` ставит `—`.
   Проверено: вопросы вроде *«What make of car did the March Hare drive?»* и *«What year did
   Harry Potter meet Alice?»* корректно получают такой отказ — с сохранением всех трёх блоков.

## Шаг 3 — Интерактивный CLI (`main_chat.py`)

```bash
python3 main_chat.py
```

Бесконечный цикл: пользователь вводит вопрос → в консоли идёт **цветное логирование этапов**
(`[STATE]` пурпур, `[REWRITE]` циан, `[RETRIEVE]` синий, `[THRESHOLD]` жёлтый, `[RERANK]`
маджента, `[CHAT]` зелёный, `[REFUSAL]` красный) → затем структурный ответ (заголовки
подсвечены), метаданные источников для проверки и панель **TASK STATE (sticky memory)**.

Команды: `/state` — показать память задачи, `/reset` — начать заново, `/exit` — выйти.

## Шаг 4 — Автотест длинных сценариев (`run_scenarios.py`)

```bash
python3 run_scenarios.py      # пишет scenario_report.md
```

Прогоняет два диалога по 13 сообщений через свежий `ChatAgent`, логирует ход и в конце по каждому
сценарию подтверждает:

1. **Теряет ли ассистент цель к 12-му сообщению?** — сверяется `TaskState.goals` на 12-м ходу.
2. **Стабильно ли выводит блоки источников и цитат?** — на каждом не-refusal ответе проверяется
   наличие `## Sources` и `## Quotes Used`.
3. **(бонус)** держится ли введённое по ходу ограничение «не упоминать X».

- *Сценарий 1* — допрос про Чеширского Кота с постоянной сменой фокуса (улыбка → исчезновение →
  …) и введённым на 6-м сообщении ограничением «не упоминать Королеву».
- *Сценарий 2* — суд над Валетом с фиксацией терминов («'the accused' = the Knave of Hearts»,
  «'the poem' = зачитанные стихи») и промежуточных выводов.

### Реальные результаты последнего прогона (см. `scenario_report.md`)

| Проверка | Сценарий 1 (Чеширский Кот) | Сценарий 2 (Суд над Валетом) |
|---|---|---|
| **Q1 — цель НЕ потеряна на 12-м сообщении** | ✅ Да (`goals` начинается с *«Learn everything about the Cheshire Cat»*) | ✅ Да (*«Work through the trial of the Knave of Hearts»*) |
| **Q2 — источники + цитаты на каждом ответе** | ✅ 13/13 | ✅ 13/13 |
| **Q3 (бонус) — ограничение удержано** | ⚠️ 6/7 ходов после правила чисто; утечка на msg 11 | — (ограничений-запретов нет) |

**Про честную утечку (msg 11).** Ограничение «не упоминать Королеву» корректно сохранено в
`TaskState`, вынесено в топ системного промпта наивысшим приоритетом и удержано на 6 из 7 ходов
после его ввода. Единственная утечка — на вопросе про спор о том, можно ли обезглавить то, у чего
нет тела: в самом тексте книги участниками именно этого спора являются палач, Король **и**
Королева, и модель, оставаясь верной первоисточнику, называет всех троих. Это реальная граница
«жёсткое ограничение vs фактологическая точность», когда пользователь спрашивает ровно про сцену,
где запрещённая сущность — прямой участник события. Не стал «подкручивать» промпт до искусственного
0 утечек: обязательные Q1/Q2 проходят на обоих сценариях чисто, а бонусный Q3 показан как есть.

> Замечание про проверку ограничения: запрет распространяется на **содержание** (блоки `## Answer`
> и `## Quotes Used`), но не на неизменяемые названия глав в блоке `## Sources` — глава VIII
> буквально называется *«The Queen's Croquet-Ground»*, и процитировать её как источник иначе
> нельзя. Поэтому валидатор `run_scenarios.py` ищет запретное слово только в Answer+Quotes.

## Запуск: окружение

```bash
# 1) Ollama (эмбеддинги запроса) — модель nomic-embed-text
ollama serve &
ollama list                       # nomic-embed-text должен быть в списке

# 2) DeepSeek API-ключ
export DEEPSEEK_API_KEY='your-key-here'

# 3) venv с openai + sentence-transformers (тот же, что для прошлого этапа)
cd week_5_RAG/chat_with_RAG
../../deepseek-env/bin/python main_chat.py        # интерактив
../../deepseek-env/bin/python run_scenarios.py    # автотесты
```

Первый прогон подтянет веса cross-encoder'а `BAAI/bge-reranker-base` (~1.1 ГБ, кэш в
`~/.cache/huggingface`); дальше он грузится один раз на процесс (ленивый singleton в
`retrieval_v2.py`). На CPU реранк ~9 с на 10 пар — поэтому полный прогон двух сценариев занимает
несколько минут.

```
request_to_RAG/         базовый RAG: embed -> cosine top-k -> DeepSeek
        │
reranking_and_rewrite/  + Query Rewrite, + broad search, + cross-encoder rerank,
        │               + жёсткий порог «не знаю», + структурный ответ (Answer/Quotes/Sources)
        │
chat_with_RAG/          + stateful-диалог: TaskState (цель/термины/ограничения) поверх всего,
                        context-aware rewrite по истории, окно истории + липкая память задачи
```
