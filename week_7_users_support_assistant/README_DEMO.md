# Демо ассистента поддержки (`demo.py`)

`demo.py` за один прогон показывает шесть аспектов ассистента под понятными заголовками. Это не
тест: скрипт ничего не утверждает, а печатает результат вместе с внутренностями — переписанным
поисковым запросом, косинусом и rerank-скором, фактами из CRM, — чтобы было видно **что** и
**почему** делает ассистент. Дёргает Python-API напрямую (не CLI через подпроцесс), поэтому
диагностика видна прямо в выводе.

## Запуск

```bash
cd /media/polurival/Data/AI-Projects/AI-Playground/week_7_users_support_assistant
PY=../deepseek-env/bin/python

$PY demo.py            # все шесть разделов
$PY demo.py A C E      # только выбранные разделы (буквы в любом порядке)
```

Предпосылки те же, что у ассистента ([README.md §1](README.md)):

- **Ollama** с `nomic-embed-text` (`sudo snap start ollama`);
- **индекс документации** построен: `$PY main.py ingest` (иначе demo сразу подскажет команду);
- **LLM** для разделов B–E: `DEEPSEEK_API_KEY` (по умолчанию) или локальная модель.

Разделы **A** и **F** работают **без LLM** — это чистый MCP. Если ключа нет и локальная модель не
поднята, запускайте `$PY demo.py A F`.

## Что показывает каждый раздел

| Раздел | Аспект | LLM |
|---|---|---|
| **A** | CRM через MCP: список тикетов + карточка клиента и тикета, без единого вызова LLM | нет |
| **B** | Ядро задания: один вопрос «Почему не работает авторизация?» × три тикета → три разных верных ответа | да |
| **C** | Контекст тикета управляет **поиском**, не только ответом: тот же вопрос с тикетом и без — разные запросы и источники | да |
| **D** | Тариф решает и вне авторизации: `API-429` на тарифе Pro (лимит 600 запросов/мин) | да |
| **E** | Вопрос вне документации → отказ по порогу rerank, LLM **не вызывается**; рядом живой вопрос для сравнения | да |
| **F** | Запись в CRM через MCP: заметка в тикет, с **откатом данных** в конце | нет |

## Разбор ключевых разделов

### A — CRM живёт за MCP

Ассистент не читает `crm_data/*.json` напрямую. `demo.py` вызывает `crm_context.list_tickets` и
`crm_context.ticket_context`, которые под капотом поднимают `crm_mcp_server.py` подпроцессом и
общаются по stdio:

```
Открытые тикеты: crm_list_tickets(status='open')  ->  MCP
  TCK-1042 | SSO-402   | USR-004 | Не могу войти через корпоративный SSO
  TCK-1043 | AUTH-403  | USR-002 | Авторизация перестала работать, пароль точно верный
  TCK-1044 | MFA-401   | USR-003 | Не проходит вход: код из приложения не принимается
  TCK-1051 | API-429   | USR-005 | Интеграция получает 429 от API
  TCK-1045 | PLAN-409  | USR-006 | Не вижу раздел «Вебхуки» в настройках
```

### B — один вопрос, три ответа

Вопрос везде «Почему не работает авторизация?». Разница — только контекст тикета из CRM. В строке
диагностики видно, как факты тикета попали в переписанный запрос:

```
TCK-1042 -> «Причина ошибки SSO-402 — ваш тариф Free ... SSO работает только на Business»
   [запрос: 'код ошибки SSO-402 раздел auth тариф Free ...' | max_rerank: 1.000 | вызов LLM: да]

TCK-1043 -> «После 5 неудачных попыток подряд аккаунт заблокирован на 30 минут (AUTH-403)»
   [запрос: 'AUTH-403 auth Pro авторизация не работает пароль верный' | max_rerank: 0.988]

TCK-1044 -> «MFA-401 — рассинхрон времени. Синхронизируйте часы на iPhone: Настройки → ...»
   [запрос: 'код ошибки MFA-401 раздел auth ... код из приложения не принимается' | max_rerank: 0.997]
```

Инструкция для TCK-1044 дана именно под iPhone, потому что в тикете `app_version: ios 4.6.9`; а
про «взлома нет» в TCK-1043 ассистент говорит по **внутренней заметке** тикета (журнал входов),
а не по документации.

### C — контекст рулит поиском

Тот же вопрос без тикета переписывается в `'авторизация не работает'` и вытягивает общую FAQ; с
тикетом TCK-1042 — в `'код ошибки SSO-402 раздел auth тариф Free ...'` и первым источником
поднимается раздел про SSO-402 (cosine 0.850, rerank 1.000). Контекст тикета меняет **вход** в
retrieval, а не только текст ответа.

### E — порог отказа

```
«Какая погода в Москве?»  [max_cosine: 0.743 | max_rerank: 0.000 | вызов LLM: НЕТ (отказ)]
«Как включить 2FA?»       [max_cosine: 0.749 | max_rerank: 0.999 | вызов LLM: да]
```

По косинусу мусор и живой вопрос неразличимы (0.743 против 0.749). Решает rerank: 0.000 против
0.999. Поэтому жёсткий порог стоит на rerank-скоре (< 0.10 → отказ), и при мусоре LLM не
вызывается вовсе. Подробнее — [README.md §5](README.md).

### F — запись в CRM, с откатом

Раздел добавляет заметку в TCK-1042 через `crm_add_ticket_note` (единственный инструмент MCP на
запись), читает тикет обратно и показывает заметку на месте, после чего **восстанавливает
`tickets.json` из бэкапа**, снятого в начале раздела. Прогон demo не оставляет следов в данных
CRM.
