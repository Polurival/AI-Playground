# SPEC — Ассистент поддержки пользователей (Week 7)

## 1. Задача

Мини-сервис поддержки пользователей, который:

1. отвечает на вопросы о продукте,
2. использует **RAG** по FAQ и документации продукта,
3. учитывает **контекст пользователя или тикета**, получаемый **через MCP** (JSON с
   пользователями/тикетами вместо реальной CRM),
4. на вопрос вида «Почему не работает авторизация?» отвечает **с учётом данных тикета**.

## 2. Проблема: нет своего продукта и нет CRM

Ни продукта, ни CRM, ни реальных пользователей нет. Решение — построить полностью
самодостаточный, воспроизводимый стенд:

| Чего нет в реальности | Чем заменяем | Почему так |
|---|---|---|
| Продукт | Вымышленный SaaS **TaskPilot** (задачи/проекты: SSO, 2FA, API-ключи, вебхуки, тарифы) | Домен, где вопрос «не работает авторизация» имеет несколько разных причин, и правильный ответ **зависит от данных тикета** — ровно то, что проверяет задание |
| FAQ и документация | `product/docs/*.md` + `product/README.md`, написаны нами на русском (~10 файлов) | Корпус мал (быстрый ingest), но покрывает те же типы источников, что и реальный: обзор, FAQ, разделы документации, справочник кодов ошибок |
| CRM | `crm_data/users.json` + `crm_data/tickets.json` | Задание прямо разрешает «или JSON с пользователями / тикетами». Данные видны глазами и правятся руками |
| CRM API | Собственный **MCP-сервер** `crm_mcp_server.py` поверх этих JSON, stdio-транспорт | Ассистент не читает JSON напрямую — он ходит в CRM исключительно через MCP-инструменты, как ходил бы в настоящую CRM |

Ключевой смысл стенда: **CRM-данные и документация специально расходятся между тикетами.**
Один и тот же вопрос «Почему не работает авторизация?» даёт три разных верных ответа в
зависимости от тикета — это и есть демонстрация «учёта контекста тикета», а не RAG-ответа
«вообще про авторизацию».

## 3. Продукт TaskPilot (вымышленный)

SaaS для управления задачами и проектами. Тарифы:

| Тариф | SSO (SAML) | API rate limit | Вебхуки | Участников |
|---|---|---|---|---|
| Free | нет | 60 req/min | нет | до 5 |
| Pro | нет | 600 req/min | да | до 50 |
| Business | да | 3000 req/min | да | без лимита |

Коды ошибок авторизации (справочник в документации):

| Код | Значение | Реальная причина |
|---|---|---|
| `AUTH-401` | неверный логин/пароль | пользователь ошибся паролем |
| `AUTH-403` | аккаунт заблокирован | 5 неудачных попыток → блок на 30 минут |
| `SSO-402` | SSO недоступен на тарифе | SSO требует Business, у пользователя Free/Pro |
| `SSO-407` | SAML-конфиг не завершён | админ не загрузил метаданные IdP |
| `MFA-401` | код 2FA не принят | рассинхрон времени TOTP |
| `API-401` | API-ключ не принят | ключ отозван или истёк |
| `API-429` | rate limit | превышен лимит тарифа |

## 4. Модель данных CRM

`crm_data/users.json` — список пользователей:

```json
{
  "user_id": "USR-004",
  "name": "Марина Ковалёва",
  "email": "m.kovaleva@vector-lab.ru",
  "org": "Vector Lab",
  "plan": "free",
  "role": "admin",
  "sso_enabled": false,
  "mfa_enabled": true,
  "seats_used": 5,
  "created_at": "2026-02-11",
  "locale": "ru"
}
```

`crm_data/tickets.json` — список тикетов:

```json
{
  "ticket_id": "TCK-1042",
  "user_id": "USR-004",
  "subject": "Не могу войти через корпоративный SSO",
  "status": "open",
  "priority": "high",
  "product_area": "auth",
  "error_code": "SSO-402",
  "app_version": "web 4.7.1",
  "created_at": "2026-07-13",
  "messages": [
    {"author": "user", "at": "2026-07-13T09:12", "text": "..."},
    {"author": "support", "at": "2026-07-13T10:02", "text": "..."}
  ]
}
```

Набор данных: ~6 пользователей, ~8 тикетов. Обязательно есть три «авторизационных» тикета с
разными причинами (`SSO-402` на Free, `AUTH-403` блокировка, `MFA-401` рассинхрон TOTP), чтобы
показать зависимость ответа от контекста.

## 5. MCP-слой (замена CRM)

Собственный MCP-сервер `crm_mcp_server.py` (FastMCP, stdio) — по образцу
`week_4_mcp/day_17_create_mcp/git_mcp_server.py`. Инструменты:

| Инструмент | Назначение |
|---|---|
| `crm_get_user(user_id)` | карточка пользователя (тариф, SSO/2FA, роль, org) |
| `crm_get_ticket(ticket_id)` | тикет целиком, включая переписку |
| `crm_list_tickets(user_id="", status="", limit=20)` | тикеты с фильтрами |
| `crm_search_tickets(query, limit=10)` | поиск по subject/тексту сообщений |
| `crm_add_ticket_note(ticket_id, note)` | записать во внутренние заметки тикета (запись, не только чтение) |

Клиент `crm_mcp_client.py` поднимает сервер подпроцессом и говорит по stdio — как
`git_mcp_client.py` в day_17. Модуль `crm_context.py` собирает из вызовов MCP компактный
текстовый блок контекста для промпта.

Требование: ассистент **не импортирует JSON напрямую**. Единственный путь к данным CRM — MCP.

## 6. RAG-слой

Тот же двухэтапный конвейер, что в `week_7_assistant` (переиспользуем модули week_5, ничего не
переписываем):

1. **rewrite** — вопрос переписывается в плотный поисковый запрос; **в rewrite подмешивается
   контекст тикета** (тема, product_area, код ошибки, тариф), поэтому «почему не работает
   авторизация» превращается в «SSO-402 SAML вход тариф Free ограничение», и retrieval попадает
   в нужную статью, а не в общую страницу про логин;
2. **embed** — `nomic-embed-text` через Ollama, префикс `search_query:` (документы на ingest —
   `search_document:`);
3. **recall** — косинус по всем чанкам, top-12; косинусный порог 0.50 работает только как
   грубый фильтр (пустой индекс, полный мусор);
4. **rerank** — cross-encoder `BAAI/bge-reranker-base`, top-4;
5. **жёсткий порог по rerank-скору** — если лучший чанк набрал < 0.10, LLM **не вызывается**:
   вопрос вне документации → отказ + предложение эскалации на инженера (в поддержке выдумывать
   ответ дороже, чем признать незнание).

   Решает именно rerank, а не косинус: на русском корпусе `nomic-embed-text` держит высокий
   базовый уровень — «Какая погода в Москве?» даёт лучший косинус **0.743** против **0.850** у
   настоящего попадания, и порог по косинусу на таком зазоре либо пропускает мусор, либо режет
   живые вопросы. Cross-encoder судит пару (запрос, чанк) напрямую и разделяет их начисто:
   **0.000** против **1.000**. Если cross-encoder не загрузился, порог не применяется, и
   конвейер продолжает работать на одном косинусе;
6. **generation** — ответ строго по выдержкам из документации + карточка пользователя + тикет,
   со ссылками на файлы документации.

## 7. Поток ответа

```
ask --ticket TCK-1042 "Почему не работает авторизация?"
  1. CRM через MCP    crm_context.py   crm_get_ticket -> crm_get_user (тариф, SSO, код ошибки)
  2. Query rewrite    assistant.py     вопрос + контекст тикета -> поисковый запрос
  3. Embed + recall   rag.py           top-12 по косинусу (грубый порог 0.50)
  4. Rerank           rag.py           cross-encoder -> top-4
  5. Жёсткий порог    rag.py           max rerank >= 0.10 ? дальше : ОТКАЗ (без вызова LLM)
  6. Generation       assistant.py     LLM(документация + пользователь + тикет)
  7. Вывод            main.py          ответ + источники + диагностика
```

## 8. Демо-сценарии (обязательный приёмочный набор)

| # | Команда | Ожидаемое поведение |
|---|---|---|
| 1 | `ask "Почему не работает авторизация?"` (без тикета) | Общий ответ по FAQ: перечисляет возможные причины, просит уточнить код ошибки |
| 2 | `ask --ticket TCK-1042 "Почему не работает авторизация?"` | Пользователь на **Free**, ошибка `SSO-402` → SSO доступен только на Business; ответ называет тариф пользователя и предлагает вход по паролю или апгрейд |
| 3 | `ask --ticket TCK-1043 "Почему не работает авторизация?"` | Ошибка `AUTH-403` → аккаунт временно заблокирован после 5 неудачных попыток, разблокировка через 30 минут / сброс пароля |
| 4 | `ask --ticket TCK-1044 "Почему не работает авторизация?"` | Ошибка `MFA-401` → рассинхрон времени TOTP, инструкция по пересинхронизации |
| 5 | `ask --ticket TCK-1051 "Почему API отдаёт 429?"` | Тариф Pro → лимит 600 req/min, из документации по API |
| 6 | `ask "Какая погода в Москве?"` | Отказ: rerank 0.000 < 0.10, LLM не вызывается |

Сценарии 2–4 — ядро задания: **один вопрос, три разных ответа, разница только в тикете.**

## 9. Интерфейс

CLI (тот же паттерн, что `week_7_assistant/main.py`):

```
python3 main.py ingest                                  # проиндексировать документацию
python3 main.py ask "вопрос"                            # без контекста
python3 main.py ask --ticket TCK-1042 "вопрос"          # с контекстом тикета
python3 main.py ask --user USR-004 "вопрос"             # с контекстом пользователя
python3 main.py                                         # REPL
```

Команды REPL: `/ticket <id>` (закрепить тикет), `/user <id>`, `/tickets [status]`, `/whoami`,
`/note <текст>` (записать заметку в тикет через MCP), `/model <local|deepseek>`, `/quit`.
Обычный текст = вопрос в текущем контексте.

## 10. Переиспользование (ничего не пишем заново)

  `retrieval_v2.rerank_with_cross_encoder`, `chat_with_RAG/llm_provider` (DeepSeek ↔ Ollama);

- `week_4_mcp/day_17_create_mcp` — образец собственного MCP-сервера и stdio-клиента.

## 11. Границы

- Не веб-UI и не HTTP-сервис — CLI (мини-сервис в терминале);
- CRM только на чтение + одна запись (`crm_add_ticket_note`); нет авторизации/мультиарендности;
- Продукт вымышленный: документация и тикеты — синтетические данные стенда, не реальный сервис.
