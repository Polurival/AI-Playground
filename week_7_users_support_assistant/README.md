# Week 7 — Ассистент поддержки пользователей (RAG + MCP + CRM)

Мини-сервис поддержки: отвечает на вопросы о продукте по его **документации и FAQ (RAG)**,
подтягивая **контекст пользователя и тикета из CRM через MCP**, ссылается на использованные
файлы документации и **отказывается отвечать**, если в документации нет ответа.

Своего продукта и CRM нет, поэтому стенд самодостаточный: вымышленный SaaS **TaskPilot**
(документация в `product/`) и синтетическая CRM (`crm_data/*.json`) за собственным MCP-сервером.
Проектирование — в [SPEC.md](SPEC.md), ход работ — в [PLAN.md](PLAN.md).

**Главное в демо:** один и тот же вопрос «Почему не работает авторизация?» даёт **три разных
верных ответа** — разница только в тикете, из которого ассистент узнаёт тариф клиента и код
ошибки (см. §4).

> Всё сразу одной командой: `../deepseek-env/bin/python demo.py` — шесть разделов с живым
> выводом. Описание — [README_DEMO.md](README_DEMO.md).

## 3. Как это работает

```
ask --ticket TCK-1042 "Почему не работает авторизация?"
  1. Контекст CRM (MCP)   crm_context.py   crm_get_ticket -> crm_get_user
  2. Перезапись запроса   assistant.py     _rewrite() — вопрос + факты тикета
  3. Эмбеддинг + recall   rag.py           nomic-embed-text, косинус top-12
  4. Rerank               rag.py           cross-encoder -> top-4
  5. Жёсткий порог        rag.py           max rerank >= 0.10 ? дальше : ОТКАЗ (LLM не вызывается)
  6. Ответ                assistant.py     LLM(выдержки + карточка клиента + тикет)
  7. Вывод                main.py          ответ + источники + диагностика
```

1. **Контекст CRM через MCP.** `crm_context.py` поднимает `crm_mcp_server.py` подпроцессом
   (stdio) и вызывает его инструменты: `crm_get_ticket` → `crm_get_user`. Ассистент **не читает
   `crm_data/*.json` напрямую** — только через MCP, как ходил бы в настоящую CRM. Инструменты
   отдают JSON: и человекочитаемый блок для промпта, и отдельные факты (тариф, код ошибки) —
   собирает `crm_context.py`.

2. **Перезапись запроса с фактами тикета.** Ключевой шаг: в поисковый запрос подмешиваются код
   ошибки, раздел продукта и тариф. Поэтому «почему не работает авторизация?» превращается в
   `код ошибки SSO-402 раздел auth тариф Free авторизация корпоративный SSO не работает` —
   и поиск попадает в раздел про SSO, а не в общую страницу про вход.

3. **Эмбеддинг и recall.** Запрос эмбеддится с префиксом `search_query:` (документы на ingest — с
   `search_document:`: `nomic-embed-text` асимметричен и требует парных префиксов), косинус по
   всем чанкам, top-12.

4. **Rerank.** `BAAI/bge-reranker-base` пересчитывает пары (запрос, чанк) напрямую, оставляет 4.

5. **Жёсткий порог по rerank-скору.** Если лучший чанк набрал < 0.10, ассистент выдаёт отказ и
   **не вызывает LLM вообще** — выдумать ответ он физически не может. Почему порог по rerank, а
   не по косинусу — см. §5.

6. **Ответ.** Выдержки + карточка клиента + переписка тикета уходят в один промпт со строгой
   системной инструкцией: отвечать по-русски, только по этим материалам, учитывать тариф,
   ссылаться на файлы. Вызов идёт через `llm_provider` — DeepSeek или локальная Ollama.

## 5. Вопросы вне документации отклоняются

```bash
$PY main.py ask "Какая погода в Москве?"
```

```
В документации TaskPilot не нашлось раздела, релевантного этому вопросу, поэтому отвечать
наугад не буду. Уточните вопрос или пришлите код ошибки с экрана — по нему причина
определяется однозначно. Могу передать обращение инженеру.

[провайдер: deepseek (deepseek-chat, cloud) | запрос: 'погода Москва' | max_cosine: 0.743 | max_rerank: 0.000 | контекст CRM: нет]
```

`max_rerank 0.000 < 0.10` → LLM **не вызывается**, выдумать ответ нечему. Если тикет закреплён,
отказ дополняется обещанием эскалации инженеру с номером обращения.

### Почему порог по rerank, а не по косинусу

В `week_7_assistant` порог стоял на косинусе (0.55) и на английском корпусе Typer работал. Здесь
он оказался мёртвым: на русском корпусе `nomic-embed-text` держит высокий базовый уровень —

| Запрос | max_cosine | max_rerank |
|---|---|---|
| «Какая погода в Москве?» (мусор) | 0.743 | **0.000** |
| «Почему не работает авторизация?» (попадание) | 0.742–0.850 | **0.993–1.000** |

По косинусу мусор и попадание неразличимы (0.743 против 0.742!), а cross-encoder, который судит
пару (запрос, чанк) напрямую, разделяет их начисто. Поэтому косинусный порог понижен до 0.50 и
оставлен грубым фильтром, а решение принимает rerank-скор. Если cross-encoder не загрузился,
порог не применяется — конвейер работает, но теряет эту защиту.

## 7. Инструменты MCP-сервера CRM

## 10. Известные особенности

- Если вопрос вне документации задан **внутри тикета**, факты тикета попадают в поисковый запрос,
  порог проходится и LLM всё же вызывается — но отвечает отказом по системной инструкции
  («могу помочь только по TaskPilot»). Отказ сохраняется, экономия на вызове LLM — нет.
- `product/` и `crm_data/` — синтетические данные стенда. Реального сервиса TaskPilot нет.
