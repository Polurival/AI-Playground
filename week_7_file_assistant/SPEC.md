# SPEC — Ассистент для работы с файлами проекта (Week 7)

## 1. Задача

Сделать ассистента, который **активно работает с файлами проекта**, а не просто отвечает
текстом. Через MCP/tooling ассистент должен уметь:

1. читать файлы проекта,
2. искать информацию по нескольким файлам,
3. анализировать содержимое,
4. создавать или изменять файлы.

Ключевые требования задания:

- ассистент **сам инициирует** работу с файлами — я задаю задачу **на уровне цели**
  («обнови changelog по изменениям в коде»), а не «открой файл X, впиши строку Y»;
- работа минимум с **2–3 файлами**;
- изменения **сохраняются** или выводятся как **diff**;
- результат **воспроизводим** повторно.

## 2. Главная архитектурная идея: это агент, а не RAG

`week_7_assistant` и `week_7_users_support_assistant` — конвейеры «вопрос → retrieval →
ответ». Здесь другое: цель формулируется абстрактно, а **какие файлы читать, что в них
искать и что править — решает сама модель**. Значит нужен **агентный цикл с вызовом
инструментов** (tool-calling loop), а не одноразовый retrieval.

```
цель пользователя
   └─► LLM планирует ─► вызывает инструмент ─► получает результат ─┐
          ▲                                                        │
          └──────────────── повторяет, пока не выполнит ◄──────────┘
   └─► финальный отчёт + сохранённые файлы / diff
```

Файлы ассистент трогает **только через инструменты** — как настоящий агент через MCP, а не
через прямой `open()` в бизнес-логике.

## 3. Проблема: `chat_completion` не поддерживает native tool-calling

Переиспользуемый `week_5_RAG/chat_with_RAG/llm_provider.chat_completion(...)` возвращает
**только строку** — без `tools` / `tool_calls`. Чтобы не переписывать провайдер, используем
**текстовый JSON-протокол вызова инструментов** (ReAct-стиль):

- системный промпт описывает доступные инструменты и требует отвечать **строго одним JSON**:
  либо `{"tool": "<имя>", "args": {...}}`, либо `{"final": "<итоговый отчёт>"}`;
- агентный цикл парсит JSON, исполняет инструмент через MCP, возвращает результат в диалог
  как `observation`, и снова спрашивает модель — до `final` или лимита шагов.

LLM для этого задания — **только DeepSeek** (cloud). Локальную модель не используем: агентный
tool-loop требует стабильного следования протоколу, а локальные модели на ноутбуке для этого
слишком слабы/медленны. Порог шагов (`max_steps`) защищает от зацикливания.

## 4. Инструменты работы с файлами (собственный MCP-сервер)

Собственный `file_mcp_server.py` (FastMCP, stdio) — по образцу
`week_4_mcp/day_17_create_mcp/git_mcp_server.py` и `crm_mcp_server.py`. Все пути —
**относительно корня проекта** (`--root`), с защитой от выхода за корень (path traversal).

| Инструмент | Тип | Назначение |
|---|---|---|
| `list_files(glob="**/*")` | чтение | список файлов проекта по glob |
| `read_file(path)` | чтение | содержимое файла (с номерами строк) |
| `search_files(pattern, glob="**/*")` | чтение | grep-поиск по нескольким файлам → `file:line: текст` |
| `analyze_project()` | чтение | сводка: языки, размеры, кол-во файлов, точки входа |
| `write_file(path, content)` | запись | создать/перезаписать файл (в пределах корня) |
| `propose_change(path, content)` | запись/dry-run | вернуть **unified diff** без записи на диск |

Клиент `file_mcp_client.py` поднимает сервер подпроцессом (stdio) — как `git_mcp_client.py`.

### 3.1 Режим записи: `--apply` vs dry-run (по умолчанию)

- **по умолчанию (dry-run)** — все изменения показываются как **unified diff** и на диск
  **не пишутся** (`write_file`/`propose_change` возвращают diff);
- **`--apply`** — изменения реально сохраняются на диск.

Так одновременно закрываются оба варианта из задания («сохраняются **или** выводятся как
diff») и обеспечивается безопасная воспроизводимость: сначала посмотрел diff, потом применил.

## 5. Целевой проект (стенд)

Как в support-ассистенте выдуман TaskPilot, здесь нужен **детерминированный проект** для
воспроизводимого демо. Кладём небольшой синтетический пакет `sample_project/` — маленькая
Python-библиотека, где один компонент используется в нескольких файлах:

```
sample_project/
├── README.md              # намеренно устаревший/неполный — его будем обновлять
├── CHANGELOG.md           # с секцией [Unreleased] — в неё генерируем запись
├── notifier/
│   ├── __init__.py
│   ├── client.py          # класс Notifier — центральный компонент
│   ├── email_channel.py   # использует Notifier
│   ├── sms_channel.py     # использует Notifier
│   └── retry.py           # утилита retry, используется в каналах
└── examples/
    └── quickstart.py      # тоже использует Notifier
```

Свойства стенда, нужные для сценариев:
- `Notifier` используется в **4+ местах** (каналы, пример) → сценарий «найти все usages»
  реально многофайловый;
- README **отстаёт** от кода (не описывает `sms_channel`) → сценарий «обновить документацию»;
- в коде есть нарушения инвариантов (модуль без docstring, публичная функция без
  docstring) → сценарий «проверка правил»;
- есть `CHANGELOG.md` с `[Unreleased]` → сценарий «сгенерировать запись changelog».

Ассистент **project-agnostic**: `--root <путь>` наводит его на любой каталог; стенд —
дефолт для воспроизводимого демо.

## 6. Сценарии (реализуем 4; задание требует ≥2)

| # | Цель (goal-level) | Что делает агент | Результат |
|---|---|---|---|
| 1 | «Найди все места, где используется компонент `Notifier`, и составь отчёт» | `search_files` + `read_file` по нескольким файлам, анализ | отчёт `file:line` + краткое описание каждого использования (в консоль или в `USAGES.md`) |
| 2 | «Проверь проект на инварианты: у каждого модуля и публичной функции есть docstring; нет оставшихся TODO» | `list_files` → `read_file` каждого → анализ | список нарушений `file:line: правило` (отчёт) |
| 3 | «Сгенерируй запись в CHANGELOG по последним изменениям кода» | `analyze_project` + `read_file` + правка `CHANGELOG.md` | обновлённая секция `[Unreleased]` — **diff** (dry-run) или запись (`--apply`) |
| 4 | «Обнови README, чтобы он описывал все каналы уведомлений» | `search_files`/`read_file` каналов → правка `README.md` | обновлённый README — **diff** или запись |

Сценарии 1 и 2 — чтение/поиск/анализ (много файлов); 3 и 4 — создание/изменение с diff.
Минимум задания (≥2 сценария, ≥2–3 файла, diff/сохранение, воспроизводимость) перекрыт.

## 7. Поток выполнения

```
do --root sample_project "Обнови CHANGELOG по изменениям кода"
  1. Старт MCP        file_mcp_client поднимает file_mcp_server (stdio)
  2. Системный промпт assistant.py   описывает инструменты + JSON-протокол + режим (apply/dry-run)
  3. Агентный цикл    agent_loop.py  LLM -> {"tool":...} -> MCP -> observation -> ... (<= max_steps)
  4. Правка файла     через write_file (apply) или propose_change (diff)
  5. Финал            LLM -> {"final": "..."} -> печать отчёта + diff/список изменённых файлов
```

## 8. Демо-набор (приёмочный)

| # | Команда | Ожидаемо |
|---|---|---|
| 1 | `do --root sample_project "найди все использования Notifier"` | таблица `file:line` по 4+ файлам + резюме, файлы не менялись |
| 2 | `do --root sample_project "проверь docstring-инварианты и TODO"` | перечень нарушений с `file:line`, файлы не менялись |
| 3 | `do --root sample_project "добавь запись в CHANGELOG"` | unified diff секции `[Unreleased]` (dry-run) |
| 4 | `do --root sample_project --apply "добавь запись в CHANGELOG"` | тот же diff + файл реально изменён на диске |
| 5 | `do --root sample_project "обнови README про каналы"` | diff README с добавленным описанием `sms_channel` |
| 6 | повтор сценария 3 после `git checkout sample_project` | идентичный diff → воспроизводимость |

## 9. Интерфейс (CLI, паттерн `week_7_assistant/main.py`)

```
python3 main.py --root sample_project do "<цель>"        # агент выполняет цель (dry-run)
python3 main.py --root sample_project do --apply "<цель>" # с записью на диск
python3 main.py --root sample_project tools               # список MCP-инструментов (проверка связи)
python3 main.py --root sample_project                     # REPL
```

REPL: обычный текст = цель; `/apply on|off` (переключить режим записи),
`/tools`, `/quit`. Печатается: пошаговый трейс вызовов
инструментов (что агент делал), затем финальный отчёт и diff/список изменённых файлов.

## 11. Границы

- CLI, не веб-UI;
- запись строго внутри `--root` (guard от path traversal); по умолчанию dry-run;
- целевой проект синтетический (стенд для воспроизводимого демо), но ассистент
  project-agnostic — наводится `--root` на любой каталог;
- не Git-операции (это делал `week_7_assistant`) — фокус на файловых операциях.
